# Backend Simulation Service Architecture

This document details the architecture and API specifications for the backend simulation service responsible for running the gravity-compensation trajectory computations.

## 1. Service Architecture

The backend is built as a cloud-native microservice using **FastAPI** (Python). It is designed to be highly concurrent, stateless, and mathematically performant.

### Core Components
1. **API Router / Controller:** Handles incoming HTTP requests, validates JSON payloads using Pydantic schemas, and routes them to the simulation manager.
2. **Simulation Manager:** Acts as an orchestrator. For each request, it instantiates the physics models and manages the execution loop over the requested duration. Since it is stateless, multiple simulations can run concurrently in parallel threads/workers.
3. **Physics Engine Module:** Contains the purely mathematical implementations of the `NumericalIntegrator` (Euler/RK4) and the `AltitudeController` (PID).
4. **Data Formatter:** Collects the raw state vectors generated by the engine at each timestep and packages them into a structured JSON time-series array.

## 2. API Endpoints

### `POST /simulate`
Triggers a new trajectory simulation synchronously and returns the full time-series dataset.

*   **Content-Type:** `application/json`
*   **Returns:** `200 OK` (JSON Trajectory Array) or `400 Bad Request` (Validation Error).

### `GET /health`
Liveness probe used by Kubernetes and Docker Compose to ensure the service is running and ready to accept connections.

*   **Returns:** `200 OK` `{"status": "healthy"}`

## 3. Data Schema

The system uses strong typing to validate inputs and define outputs.

### Input Parameters Schema (JSON)
Passed in the body of the `POST /simulate` request.

```json
{
  "mass": 1.0,                 // (float) Mass of the object in kg
  "target_altitude": 10.0,     // (float) Desired hover altitude in meters
  "kp": 5.0,                   // (float) Proportional gain for PID
  "ki": 0.5,                   // (float) Integral gain for PID
  "kd": 10.0,                  // (float) Derivative gain for PID
  "duration": 15.0,            // (float) Total simulation time in seconds
  "dt": 0.02,                  // (float) Integration step size in seconds
  "drag_coeff": 0.47,          // (float) Aerodynamic drag coefficient
  "reference_area": 0.01,      // (float) Cross-sectional area in m^2
  "integration_method": "rk4"  // (string) "rk4" or "euler"
}
```

### Output Trajectory Schema (JSON)
The response from the `POST /simulate` endpoint is an object containing a `trajectory` array. Each element in the array represents the state of the system at a specific timestamp.

```json
{
  "trajectory": [
    {
      "time": 0.0,                 // (float) Elapsed time (s)
      "x": 0.0,                    // (float) Local East distance (m)
      "y": 0.0,                    // (float) Local North distance (m)
      "z": 0.0,                    // (float) Local Altitude (m)
      "vx": 0.0,                   // (float) East velocity (m/s)
      "vy": 0.0,                   // (float) North velocity (m/s)
      "vz": 0.0,                   // (float) Vertical velocity (m/s)
      "f_up": 9.81,                // (float) Upward compensation force applied (N)
      "altitude_error": 10.0,      // (float) Difference from target (m)
      "kinetic_energy": 0.0,       // (float) Current KE (Joules)
      "potential_energy": 0.0,     // (float) Current PE (Joules)
      "total_energy": 0.0,         // (float) KE + PE (Joules)
      "energy_error": 0.0          // (float) Check for physical conservation (Joules)
    },
    // ... subsequent timesteps
  ]
}
```

## 4. Ability to Run Multiple Simulations

To satisfy the requirement for running multiple parallel simulations (e.g., for parameter sweeps or multiple users):

*   **Stateless Execution:** The `POST /simulate` endpoint does not store global state. All variables (integrator instances, PID states, and trajectory lists) are instantiated locally within the scope of the request handler.
*   **Asynchronous I/O via Uvicorn/FastAPI:** While the physics loop is CPU-bound, deploying the FastAPI app with multiple Uvicorn workers (e.g., `uvicorn main:app --workers 4`) allows the operating system to distribute distinct simulation requests across multiple CPU cores simultaneously.
*   **Kubernetes Scalability:** For large-scale batch processing, the `trajectory-backend` Kubernetes Deployment is configured with `replicas: N`. The Kubernetes Service acts as a load balancer, seamlessly distributing incoming POST requests across available pod instances.
