# 3D Visualization System for Trajectory Animation

This document outlines the architecture and logic for the 3D frontend visualization system, which renders the telemetry data generated by the physics backend onto a virtual 3D globe or spatial scene.

## 1. System Requirements Met

*   **Render Trajectory Path:** The full historical trajectory is drawn as a continuous polyline.
*   **Animate Object Over Time:** A 3D mesh (representing the aerospace body) moves along the path.
*   **Sync Animation:** The object's position is strictly tied to the simulated timestamps, ensuring velocity changes (acceleration/deceleration) are visually accurate.
*   **Display Metrics:** A live dashboard overlays the 3D scene, showing current altitude, velocity magnitude, and elapsed time.

## 2. Data Mapping (Backend to Frontend)

The backend returns a JSON array of trajectory points containing local ENU coordinates ($x, y, z$) and elapsed simulation time ($t$).

### Coordinate Transformation Mapping
Most 3D rendering engines (like Three.js) use a Cartesian coordinate system where the Y-axis is "Up". However, in aerospace physics, the Z-axis is traditionally "Up" (Altitude).
Therefore, upon receiving the JSON payload, the frontend must map the coordinates to the rendering engine's space:

*   **Engine X** $\leftarrow$ Backend X (East)
*   **Engine Y** $\leftarrow$ Backend Z (Altitude)
*   **Engine Z** $\leftarrow$ Backend -Y (South)

If integrating with a true globe-based engine (like CesiumJS or Three.js with a spherical projection), the ENU coordinates must first be converted to Geodetic coordinates (Latitude, Longitude, Altitude) using the WGS84 ellipsoid transformation pipeline (detailed in the Coordinate Transformations spec).

## 3. Animation Logic

To ensure the 3D object moves smoothly and accurately reflects the physics simulation, the animation loop must be decoupled from the frame rate but strictly synced to the simulation timestamps.

### Syncing with Simulation Timestamps
The animation relies on the browser's high-resolution timer (`performance.now()`) inside a rendering loop (e.g., `requestAnimationFrame` or `useFrame` in React Three Fiber).

1.  **Initialization:** When the play button is pressed, record `start_time = performance.now()`.
2.  **Tick:** On every frame, calculate the elapsed physical time: 
    $t_{elapsed} = (\text{performance.now()} - start\_time) \times \text{time\_scale}$
3.  **Interpolation:** 
    *   Find the two points in the backend JSON array where $t_k \le t_{elapsed} < t_{k+1}$.
    *   Perform a linear (or spherical linear) interpolation between the position vectors at $t_k$ and $t_{k+1}$ based on the exact fraction of time elapsed.
    *   Update the 3D mesh position to the interpolated coordinate.
4.  **Metric Updates:** Extract the velocity and altitude from the interpolated state and update the HTML overlay.

### Pseudocode for Animation Loop
```javascript
function renderFrame(currentTime) {
    let elapsedSimTime = (currentTime - startTime) * timeScale;
    
    // Find the correct segment in the trajectory array
    let index = trajectoryData.findIndex(p => p.time >= elapsedSimTime);
    
    if (index > 0 && index < trajectoryData.length) {
        let p1 = trajectoryData[index - 1];
        let p2 = trajectoryData[index];
        
        // Calculate interpolation factor (0.0 to 1.0)
        let factor = (elapsedSimTime - p1.time) / (p2.time - p1.time);
        
        // Interpolate position
        mesh.position.x = lerp(p1.x, p2.x, factor);
        mesh.position.y = lerp(p1.z, p2.z, factor); // Z is altitude in physics
        mesh.position.z = lerp(-p1.y, -p2.y, factor);
        
        // Update UI Metrics
        updateDashboard({
            altitude: lerp(p1.z, p2.z, factor),
            speed: lerp(magnitude(p1.vx, p1.vy, p1.vz), magnitude(p2.vx, p2.vy, p2.vz), factor),
            time: elapsedSimTime
        });
    } else if (index === -1) {
        // End of simulation reached
        stopAnimation();
    }
    
    requestAnimationFrame(renderFrame);
}
```

## 4. Performance Considerations

Rendering a high-frequency simulation (e.g., $dt = 0.01s$ over 60 seconds yields 6,000 points) can cause performance bottlenecks in the browser.

*   **Line Geometry Decimation:** Drawing a polyline with 6,000 vertices is expensive. The frontend should downsample the trajectory array for the static line drawing (e.g., drawing every 10th point) while keeping the full array in memory for the precise mesh animation.
*   **Instanced Rendering:** If simulating a swarm of objects, use `InstancedMesh` (in Three.js) rather than creating distinct mesh objects, allowing the GPU to draw thousands of objects in a single draw call.
*   **Separation of React and Canvas:** To prevent the entire DOM tree from re-rendering 60 times a second, metric updates should be pushed to refs or localized state components that do not trigger top-level React re-renders.
